## Findings

### âś… POSITIVE â€” what we fixed

- **[POSITIVE] Multi-query votes now aggregate correctly.** Before: `search_votes` was always 2 (1 engine + 1 query) even when the same URL was found via 5 different (query, engine) pairs. After: `search_votes = len(engines) + len(queries)` reflects true cross-source agreement. **Improves top-1 quality** (Phase 5 will use this for ranking).
- **[POSITIVE] Provenance preserved.** `fr["raw_urls"]` keeps the original URL(s) with tracking params, so callers (or future audit tools) can still see where the canonical came from. No data loss.
- **[POSITIVE] No new public API.** Function signatures of `deep_research()` unchanged. Only `fr` (per-source dict) gained one new key (`raw_urls`). Existing callers iterate by specific keys, none are broken.
- **[POSITIVE] No new dependencies.** Edit uses only stdlib (`setdefault`, `set.add`, `sorted`, `canonical_url` which was already in the file).

### âš ď¸Ź RESIDUAL

- **[LOW] `seen_urls` set is now slightly redundant.** Edit A uses it only to track first-seen (FIFO ordering for `top_urls`). `all_search` (L808) already carries every hit including duplicates. The set is small and the cost is negligible, so we keep it for clarity. If we wanted to remove it, `top_urls` could iterate `sources_meta.keys()` directly, but then we'd lose insertion order from `all_search` (Python 3.7+ dicts preserve order, so it would still work). **Action: none, tracking for future cleanup.**
- **[LOW] `raw_urls` is a new field on `fr`.** Verified in audit: no caller in `hermes_deepresearch.py` iterates `fr.keys()`. `verify_sources()` consumes specific keys (no `raw_urls`). **Risk: very low.**
- **[LOW] `meta["ranks"]` may now contain ranks from different queries for the same canonical URL.** This is intentional â€” the rank from query A and rank from query B are both valid signals. We do not use `ranks` for any current decision in Phase 2 (it's stored for future Phase 5 ranking). The 16 `test_canonical_url` tests are unaffected. **Action: tracked for Phase 5.**
- **[NOTE] The aggregator now adds `""` (empty string) to `meta["engines"]` if a search result has no `engine` field.** This is a pre-existing behaviour, not introduced by Phase 2. It means `len(meta["engines"])` could be 2 (1 real engine + 1 empty string) for a URL found by one real engine. The `test_no_duplicate_sources_after_dedup` test uses 3 real engines so it doesn't expose this. **Action: tracked for Phase 5 (when ranking uses `search_votes`).**
- **[NOTE] Test mocks pass `engine="bing"` etc. explicitly.** The real `web_search()` (in `hermes_searxng.py`, not modified in Phase 2) is responsible for adding `engine` to results. If a future engine change omits `engine`, the aggregator will count it as `""` â€” see above. **Out of Phase 2 scope.**

### Not changed (still vulnerable / broken, deferred)

- **[HIGH] `meta["engines"]` counting `""` as a unique engine** (see NOTE above). Defer to Phase 5 ranking where this matters.
- **[HIGH] `_search_result_score` ranking uses SearXNG rank only, not `search_votes`.** A URL found 5 times is ranked equal to a URL found once. **Defer to Phase 5.**
- **[MEDIUM] `top_urls` are still ordered by SearXNG rank (after `_search_result_score` sort), not by `search_votes`.** Phase 2 fixes the data, Phase 5 fixes the order. Plan says so.
- **[MEDIUM] `reformulate()` is a placeholder dict-based translator** (see L767-778 of `hermes_deepresearch.py`). Phase 2 doesn't touch this. **Defer to Phase 5 (or a separate translation step).**
