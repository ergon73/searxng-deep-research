## Findings

### ✅ POSITIVE — problems we fixed

- **[POSITIVE] No more shared secret in INSTALL.md.** Old snippet hardcoded `SEARXNG_SECRET: "18948eb4d5e7eb2c3b81e81d147919e8a7c7495b4026b5d283240cea85a82074"` (someone's real secret leaked into instructions). Replaced with `${SEARXNG_SECRET:?...}` env var pattern. **Eliminated leak-by-instruction.**
- **[POSITIVE] Fail-fast on missing secret.** `${SEARXNG_SECRET:?set SEARXNG_SECRET in .env_llm}` makes Docker Compose refuse to boot SearXNG without a real secret, instead of silently running with the placeholder. Defense-in-depth.
- **[POSITIVE] Secret moved out of `settings.yml`.** The placeholder is gone, file is empty string, env override is the only path. A naive `git grep secret_key` no longer reveals the secret.
- **[POSITIVE] `.env_llm` permissions preserved (mode 0600).** After T3 modification, file is still 0o600. `os.stat()` verified.

### ⚠️ RESIDUAL

- **[MEDIUM] `.env_llm` not in `.gitignore`.** The repo is not a git repo (verified, no `.git` directory), so this is moot *for now*. If anyone runs `git init` here, the secret is one `git add .` away from history. **Recommendation:** create a `.gitignore` that includes `.env_llm`, `.env_proxy`, and `.hermes/plans/`. Out of Phase 1 scope, but flag for Phase 8 (docs sync) or a one-line patch right now.
- **[MEDIUM] `.env_llm.example` should not contain a real key.** I did not check this file's contents in Phase 1, only that `.env_llm` exists with mode 600. The `.example` file is typically checked in and should have placeholder values, not real keys. **Action:** verify in T6.
- **[LOW] INSTALL.md L107 `privaccheck: 0`.** The text contains a sanitizer-artefact (`privaccheck` instead of `private_mode` / `private_instance`). This is a Hermes display-sanitizer bug (per runbook pitfall 18), not a security issue, but it teaches the user wrong. Defer to Phase 8.
- **[LOW] INSTALL.md L114 `secret_key: "СЮДА_ВСТАВЬ_СВОЙ_СЕКРЕТ"`.** Old instruction tells the user to put the secret in `settings.yml` instead of `.env_llm`. We added a new "Шаг 3a" that supersedes this, but the old instruction is still on the page and contradicts the new one. **Action:** defer to Phase 8 (or: do a one-line edit in T4 retroactively). Minor.
- **[LOW] `tests/test_compose_config.py::test_env_llm_has_searxng_secret` reads `.env_llm` plaintext.** This is a test file, mode 600 (write_file default), but the secret is in test memory. If pytest output is ever logged to a public CI, the value could leak via `-v` flag assertion messages. Currently the test only asserts the key is *present*, not the value, so the secret never appears in assertion output. Safe as-is. **Action:** none, just keep the test free of `repr(secret_value)` patterns.
- **[LOW] LLM_MODEL defaulted to free tier.** `LLM_MODEL=meta-llama/llama-3.1-8b-instruct:free` is the free OpenRouter model. Fine for dev, but LLM verification of facts is noisy on 8B (DR-05062026 §P1.10 flags this). Not a security issue, but a quality/ROO issue. Defer to Phase 4.
- **[NOTE] `test_settings_volume_uses_env_path_or_absolute` accepts both `${VAR}` and absolute paths.** Current code uses `${SEARXNG_SETTINGS_PATH:-./searxng/settings.yml}`. The `:-...` form means "use default if VAR is unset", which keeps the historical cwd-relative behaviour. This is composability over safety — a user could intentionally override `SEARXNG_SETTINGS_PATH=/etc/searxng-prod.yml` to mount a different config. Feature, not bug.
